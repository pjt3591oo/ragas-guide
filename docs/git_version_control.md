# Git 버전 관리 가이드

## Git이란

Git은 분산 버전 관리 시스템(DVCS)으로, 소스 코드의 변경 이력을 추적하고
여러 개발자가 동시에 작업할 수 있게 해줍니다.

리누스 토르발스가 리눅스 커널 개발을 위해 2005년에 만들었으며,
현재 소프트웨어 개발에서 사실상의 표준 버전 관리 도구입니다.

분산 버전 관리의 핵심 특징:
- 각 개발자가 저장소의 전체 복사본을 로컬에 보유
- 네트워크 연결 없이도 커밋, 브랜치, 히스토리 조회 가능
- 중앙 서버 장애 시에도 로컬 저장소로 복구 가능

## 기본 명령어

Git의 기본 워크플로우는 작업 디렉토리, 스테이징 영역(Index), 로컬 저장소의
세 영역을 기반으로 합니다.

초기 설정:
- git init: 새로운 Git 저장소 초기화
- git clone: 원격 저장소를 로컬로 복제

변경사항 관리:
- git status: 작업 디렉토리의 현재 상태 확인
- git add: 변경된 파일을 스테이징 영역에 추가
- git commit: 스테이징된 변경사항을 저장소에 기록
- git diff: 변경사항의 상세 내용 확인

원격 저장소 연동:
- git remote: 원격 저장소 관리
- git push: 로컬 커밋을 원격 저장소에 업로드
- git pull: 원격 저장소의 변경사항을 가져와서 병합
- git fetch: 원격 저장소의 변경사항을 가져오기만 함 (병합하지 않음)

## 브랜치 전략

브랜치(Branch)는 독립적인 작업 흐름을 만들어 병렬 개발을 가능하게 합니다.
Git의 브랜치는 매우 가볍고 빠르게 생성/전환할 수 있습니다.

브랜치 관련 명령어:
- git branch: 브랜치 목록 확인 및 생성
- git checkout / git switch: 브랜치 전환
- git merge: 다른 브랜치의 변경사항을 현재 브랜치에 병합
- git rebase: 커밋 히스토리를 선형으로 정리

주요 브랜치 전략:

Git Flow:
- main(master): 배포 가능한 안정 버전
- develop: 개발 통합 브랜치
- feature/*: 기능 개발 브랜치
- release/*: 배포 준비 브랜치
- hotfix/*: 긴급 수정 브랜치
- 장점: 릴리즈 주기가 명확한 프로젝트에 적합
- 단점: 복잡하고 브랜치가 많아질 수 있음

GitHub Flow:
- main 브랜치와 기능 브랜치만 사용
- 기능 브랜치에서 개발 → Pull Request → 코드 리뷰 → main에 병합
- 장점: 단순하고 CI/CD와 잘 맞음
- 단점: 복잡한 릴리즈 관리에는 부족

Trunk-Based Development:
- 짧은 수명의 기능 브랜치를 사용하거나 main에 직접 커밋
- 피처 플래그(Feature Flag)로 미완성 기능 관리
- 장점: 통합 빈도가 높아 충돌 최소화
- 단점: 성숙한 CI/CD 파이프라인 필요

## 커밋 관리

좋은 커밋은 코드 히스토리의 가독성과 유지보수성에 직접적인 영향을 미칩니다.

커밋 메시지 작성 규칙:
- 제목은 50자 이내, 명령형으로 작성 ("Add feature"가 "Added feature"보다 적합)
- 제목과 본문 사이에 빈 줄 추가
- 본문에 변경 이유와 맥락을 설명
- Conventional Commits 형식 권장: feat:, fix:, docs:, refactor:, test: 등

커밋 단위:
- 하나의 커밋은 하나의 논리적 변경만 포함
- 너무 큰 커밋은 리뷰와 되돌리기가 어려움
- 너무 작은 커밋은 히스토리를 불필요하게 복잡하게 만듦
- "이 커밋을 되돌릴 때 어떤 일이 일어나는가?"를 기준으로 판단

히스토리 정리:
- git rebase -i: 대화형 리베이스로 커밋 합치기(squash), 수정, 재정렬
- git commit --amend: 마지막 커밋 수정 (아직 push하지 않은 경우)
- git cherry-pick: 특정 커밋만 다른 브랜치에 적용

## 충돌 해결

여러 브랜치에서 같은 파일의 같은 부분을 수정하면 충돌(conflict)이 발생합니다.

충돌 해결 과정:
1. git merge 또는 git rebase 시 충돌 발생
2. 충돌 파일에 마커가 표시됨 (<<<<<<, =======, >>>>>>>)
3. 수동으로 원하는 내용으로 수정
4. git add로 해결된 파일을 스테이징
5. git commit (merge) 또는 git rebase --continue (rebase)

충돌 예방 전략:
- 자주 원격 저장소의 변경사항을 가져오기 (git pull / git fetch)
- 작은 단위로 자주 병합
- 같은 파일을 여러 사람이 동시에 수정하지 않도록 업무 분담
- 자동 포맷팅 도구 사용으로 스타일 차이에 의한 충돌 방지

## .gitignore 설정

.gitignore 파일은 Git이 추적하지 않을 파일과 디렉토리를 지정합니다.

일반적으로 무시해야 하는 파일:
- 빌드 결과물: dist/, build/, *.pyc, __pycache__/
- 의존성 디렉토리: node_modules/, .venv/, vendor/
- 환경 설정: .env, *.local
- IDE 설정: .vscode/, .idea/, *.swp
- OS 파일: .DS_Store, Thumbs.db
- 로그 파일: *.log

패턴 문법:
- *: 모든 문자 (경로 구분자 제외)
- **: 모든 디렉토리 (재귀적)
- ?: 한 문자
- !: 무시 규칙 예외 (이전에 무시된 파일을 다시 추적)
- /로 시작: 현재 디렉토리만 해당
- /로 끝남: 디렉토리만 해당

## Git Hooks

Git Hooks는 특정 Git 이벤트(커밋, 푸시 등)에 자동으로 실행되는 스크립트입니다.
.git/hooks/ 디렉토리에 위치합니다.

주요 클라이언트 훅:
- pre-commit: 커밋 전에 실행. 코드 린팅, 포맷팅 검사
- commit-msg: 커밋 메시지 검증. 형식 규칙 강제
- pre-push: 푸시 전에 실행. 테스트 실행

주요 서버 훅:
- pre-receive: 푸시를 받기 전에 실행. 브랜치 보호 규칙 적용
- post-receive: 푸시 완료 후 실행. CI/CD 트리거, 알림 전송

Husky(JavaScript), pre-commit(Python) 등의 도구를 사용하면
팀 전체에서 일관된 훅을 관리할 수 있습니다. 이들은 .git/hooks/가 아닌
프로젝트 설정 파일로 훅을 정의하여 버전 관리에 포함됩니다.
