# Docker와 컨테이너 가이드

## 컨테이너란

컨테이너(Container)는 애플리케이션과 그 실행에 필요한 모든 의존성을 하나의 패키지로
묶어 어디서든 동일하게 실행할 수 있게 하는 격리된 환경입니다.

가상 머신(VM)과의 차이:
- VM: 하이퍼바이저 위에 게스트 OS 전체를 실행. 무겁고 시작이 느림
- 컨테이너: 호스트 OS의 커널을 공유. 가볍고 수초 내 시작 가능

컨테이너는 리눅스 커널의 네임스페이스(namespace)와 cgroup(control group) 기술을
활용하여 프로세스, 네트워크, 파일시스템을 격리합니다.

## Docker 기본 개념

Docker는 컨테이너를 쉽게 생성, 배포, 실행할 수 있게 해주는 플랫폼입니다.

핵심 구성 요소:
- Docker Engine: 컨테이너를 생성하고 관리하는 런타임
- Docker Image: 컨테이너를 만들기 위한 읽기 전용 템플릿. 레이어 구조
- Docker Container: 이미지를 실행한 인스턴스. 독립된 파일시스템과 네트워크
- Docker Registry: 이미지를 저장하고 공유하는 저장소 (Docker Hub 등)

이미지와 컨테이너의 관계는 클래스와 인스턴스의 관계와 유사합니다.
하나의 이미지로 여러 개의 컨테이너를 생성할 수 있습니다.

## Dockerfile 작성

Dockerfile은 이미지를 빌드하기 위한 명령어를 담은 텍스트 파일입니다.

주요 명령어:
- FROM: 베이스 이미지 지정. 모든 Dockerfile은 FROM으로 시작
- WORKDIR: 작업 디렉토리 설정
- COPY: 호스트의 파일을 이미지에 복사
- RUN: 빌드 시 명령어 실행 (패키지 설치 등)
- ENV: 환경 변수 설정
- EXPOSE: 컨테이너가 사용할 포트 문서화
- CMD: 컨테이너 시작 시 실행할 기본 명령어
- ENTRYPOINT: 컨테이너 시작 시 항상 실행할 명령어

Dockerfile 최적화 팁:
- 레이어 캐싱을 활용하여 빌드 속도 향상. 변경이 적은 명령어를 위에 배치
- 멀티스테이지 빌드로 최종 이미지 크기 최소화
- .dockerignore 파일로 불필요한 파일 제외
- 가능하면 경량 베이스 이미지 사용 (alpine, slim 등)

## Docker Compose

Docker Compose는 여러 컨테이너로 구성된 애플리케이션을 정의하고 실행하는 도구입니다.
docker-compose.yml 파일에 서비스, 네트워크, 볼륨을 선언적으로 정의합니다.

docker-compose.yml의 주요 섹션:
- services: 각 컨테이너의 설정 (이미지, 포트, 환경 변수 등)
- volumes: 데이터 영속화를 위한 볼륨 정의
- networks: 서비스 간 통신을 위한 네트워크 정의

주요 명령어:
- docker compose up: 모든 서비스 시작 (-d 옵션으로 백그라운드 실행)
- docker compose down: 모든 서비스 중지 및 제거
- docker compose logs: 서비스 로그 확인
- docker compose ps: 실행 중인 서비스 상태 확인
- docker compose exec: 실행 중인 컨테이너에서 명령어 실행

## 볼륨과 네트워크

Docker 볼륨(Volume)은 컨테이너의 데이터를 영속적으로 저장하는 메커니즘입니다.
컨테이너가 삭제되어도 볼륨의 데이터는 유지됩니다.

볼륨 유형:
- Named Volume: Docker가 관리하는 볼륨. 권장 방식
- Bind Mount: 호스트의 특정 경로를 컨테이너에 마운트
- tmpfs Mount: 메모리에만 데이터 저장. 컨테이너 종료 시 삭제

Docker 네트워크는 컨테이너 간 통신을 관리합니다.

네트워크 드라이버:
- bridge: 기본 네트워크. 같은 호스트의 컨테이너 간 통신
- host: 호스트의 네트워크를 직접 사용. 포트 매핑 불필요
- overlay: 여러 Docker 호스트에 걸쳐 통신 (Swarm 모드)
- none: 네트워크 비활성화

## 컨테이너 보안

컨테이너 보안을 위한 모범 사례:

이미지 보안:
- 공식 이미지 또는 신뢰할 수 있는 이미지 사용
- 이미지 취약점 스캔 (Trivy, Snyk 등)
- 최소한의 패키지만 포함 (불필요한 도구 제거)
- 이미지 태그를 latest 대신 특정 버전으로 지정

런타임 보안:
- root 사용자 대신 비특권 사용자로 실행 (USER 명령어)
- 읽기 전용 파일시스템 사용 (--read-only)
- 리소스 제한 설정 (CPU, 메모리 limit)
- 불필요한 커널 기능(capabilities) 제거

비밀 정보 관리:
- 환경 변수에 비밀번호를 직접 넣지 않음
- Docker Secrets 또는 외부 비밀 관리 도구 사용 (Vault 등)
- 빌드 시 비밀 정보가 이미지 레이어에 남지 않도록 주의

## 컨테이너 오케스트레이션

여러 컨테이너를 대규모로 관리하기 위해 오케스트레이션 도구가 필요합니다.

Kubernetes(k8s):
- 컨테이너화된 애플리케이션의 배포, 스케일링, 관리를 자동화
- Pod: 컨테이너의 최소 배포 단위
- Service: Pod에 대한 안정적인 네트워크 엔드포인트
- Deployment: Pod의 선언적 업데이트 관리
- Ingress: 외부 트래픽을 클러스터 내부로 라우팅

Docker Swarm:
- Docker 내장 오케스트레이션 도구
- Kubernetes보다 단순하지만 기능이 제한적
- 소규모 환경에서 빠르게 시작할 수 있음

오케스트레이션이 제공하는 핵심 기능:
- 서비스 디스커버리와 로드 밸런싱
- 자동 스케일링 (부하에 따라 컨테이너 수 조절)
- 자가 치유 (비정상 컨테이너 자동 재시작)
- 롤링 업데이트와 롤백
- 설정 및 비밀 정보 관리
